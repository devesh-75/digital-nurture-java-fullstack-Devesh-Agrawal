Exercise 1: Inventory Management System

1. Why data structures and algorithms are essential in handling large inventories

A warehouse inventory can contain thousands or millions of distinct products, and the system must support frequent additions, updates, lookups, and deletions in real time. Without a well-chosen data structure, operations such as finding a product or updating its stock level could require scanning the entire inventory, which becomes far too slow as the inventory grows. Data structures organize information so that these operations can be performed in constant or logarithmic time rather than linear time, and algorithms provide systematic, efficient procedures for searching, sorting, and updating that data. Together they keep the system responsive and scalable as the business grows.

2. Suitable data structures for this problem

An ArrayList is useful when the inventory needs to preserve insertion order or be iterated frequently, but lookups by ID require a linear scan unless the list is sorted. A HashMap keyed by productId is generally the better choice for this scenario because it provides average O(1) time for insertion, lookup, update, and deletion, which is critical when the inventory is large and operations are frequent. A TreeMap could be used instead of a HashMap if the products need to remain sorted by ID at all times, at the cost of O(log n) operations instead of O(1).

3. Time complexity analysis

Using a HashMap<String, Product> keyed by productId:
- Add: O(1) average case, O(n) worst case if many hash collisions occur.
- Update: O(1) average case, since it is a lookup followed by a direct field modification.
- Delete: O(1) average case, since removal from a hash map does not require shifting elements.

If an ArrayList were used instead, add would be O(1) amortized (appending to the end), but update and delete would be O(n) because finding the correct product first requires a linear scan.

4. Optimizing these operations

The HashMap already gives near constant time operations for the common case. To optimize further, the initial capacity and load factor of the HashMap can be tuned to reduce the frequency of resizing when the number of products is known in advance. If searches by attributes other than productId (such as productName or category) are frequent, additional secondary indexes (extra HashMaps mapping those attributes to product IDs) can be maintained so those lookups also run close to O(1) instead of requiring a full scan. Finally, concurrent access in a real warehouse system could be optimized using a ConcurrentHashMap to avoid locking the entire structure during simultaneous reads and writes.
